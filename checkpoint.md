## 1. 프로젝트 한 줄 요약

현재 저장소는 **Team Blackbox**라는 이름의 **팀 프로젝트 기여도 자동 증빙 플랫폼**을 목표로 진행 중인 캡스톤 프로젝트로 판단된다.

핵심 방향은 아래 3가지를 하나의 흐름으로 묶는 구조다.
- 프로젝트/팀 관리
- 활동 로그 수집 및 점수화
- 교수/팀장 관점에서 기여도와 이상 징후를 확인하는 대시보드

---

## 2. 현재 진행 상태 총평

### 현재 판단
**MVP 핵심 골격은 상당 부분 구현된 상태**다.

단순 기획 단계는 이미 지났고,
- 인프라
- 백엔드 기본 도메인
- 프론트 핵심 화면
- 점수/경보/파일 관리 흐름
까지 연결되는 형태가 저장소에 존재한다.

반면 아직 마무리되지 않았거나, 저장소만으로는 검증되지 않은 영역도 분명하다.
- 운영 배포 안정화
- 프론트/백 통합 검증
- 외부 연동(GitHub/Google Drive)
- AI 분석기
- 문서와 실제 구현 상태 동기화

즉, 현재 위치는 **“MVP 후반부 ~ 데모 안정화 직전”**으로 보는 것이 가장 적절하다.

## 3. 확인된 구현 범위

### 3-1. 인프라/실행 환경
확인된 항목
- `docker-compose.yml` 존재
- PostgreSQL / Redis / Spring Boot / Next.js / Nginx 5개 서비스 구성이 잡혀 있음
- 업로드 볼륨, DB 영속 볼륨, 환경변수 구조 포함
- `docker-compose.prod.yml`도 존재하여 프로덕션 오버라이드 구조가 일부 준비되어 있음

정리
- 로컬 데모/통합 실행을 염두에 둔 구조는 이미 갖춰져 있다.
- 운영 배포 완결 여부는 별도 실행 검증이 필요하다.

### 3-2. 백엔드 기반
확인된 항목
- `backend/build.gradle` 존재
- Spring Boot 3.4.4 기반
- Java 25 toolchain 사용
- PostgreSQL, Flyway, Redis, Spring Security, Validation, JWT 의존성 포함
- `backend/src/main/resources/application.yml` 존재
- DB/JPA/Flyway/Redis/JWT/파일업로드/GitHub/Google/Claude 설정 키가 정의되어 있음

정리
- 백엔드가 단순 목업이 아니라, 실제 서비스 연결을 전제로 구성된 상태다.
- 외부 연동용 설정 키까지 이미 잡혀 있어 이후 확장 방향도 명확하다.

### 3-3. 데이터베이스 스키마
확인된 항목
- Flyway 마이그레이션 구조가 존재
- `V7__init_scores_alerts.sql` 확인
- 점수/경보/가중치/변경 로그 테이블까지 설계됨

정리
- 단순 CRUD 앱이 아니라, **기여도 산정과 이상 징후 탐지**까지 포함한 데이터 모델이 이미 설계되어 있다.

### 3-4. 프론트엔드 구조
확인된 항목
- `frontend/package.json` 존재
- Next.js 14, React 18, TypeScript, Tailwind, Zustand, Axios 사용
- dnd-kit, Recharts 포함

정리
- 프론트는 “대시보드형 협업 서비스”에 맞는 스택으로 구성되어 있다.
- 칸반/차트/상태 관리까지 고려된 상태다.

## 4. 프론트 기능별 진행 현황

### 4-1. 인증
확인된 항목
- 로그인 페이지 존재
- `auth` store 존재
- accessToken / refreshToken 저장 및 `/auth/me` 조회 흐름 구현
- Axios 인터셉터에서 401 시 refresh 처리 로직 구현

판단
- 인증 UX의 기본 흐름은 이미 구현되어 있다.
- 단, 실제 보안/예외 처리 완성도는 실행 검증이 필요하다.

### 4-2. 프로젝트 관리
확인된 항목
- `project` store 존재
- 프로젝트 목록/상세/멤버/생성/참여/동의 갱신 API 호출 구조 존재

판단
- 팀 프로젝트 단위로 서비스를 사용하는 핵심 구조는 갖춰져 있다.

### 4-3. 칸반 보드
확인된 항목
- `frontend/src/app/(dashboard)/projects/[projectId]/board/page.tsx` 존재
- 3단 컬럼(TODO / IN_PROGRESS / DONE)
- 드래그 앤 드롭
- 태스크 생성
- 담당자/우선순위/태그 필터
- 삭제 기능
- `task` store 존재

판단
- 칸반 기능은 MVP 수준을 넘어서 **실사용 가능한 형태에 가까움**.
- 프로젝트 운영 화면 중 진척도가 높은 편이다.

### 4-4. 회의록/체크인
확인된 항목
- 회의 목록/상세 화면 존재
- 체크인 코드 기반 체크인 모달 존재
- 회의록/결정사항 저장 기능 존재
- 결정사항을 태스크로 변환하는 흐름 구현

판단
- 이 기능은 단순 기록이 아니라 **회의 → 실행 태스크 연결**까지 잡혀 있어 제품 방향성과 잘 맞는다.

### 4-5. 활동 타임라인
확인된 항목
- `timeline/page.tsx` 존재
- 플랫폼/GitHub/Drive/수동 소스 구분 UI 존재
- COMMIT, PR, DOC_EDIT 등 다양한 액션 타입 표시 구조 존재
- `activity` store 존재

판단
- 활동 로그를 한 화면에 모아보는 구조는 이미 들어가 있다.
- 다만 외부 연동 데이터가 실제로 유입되는지는 별도 확인이 필요하다.

### 4-6. 파일 관리(Hash Vault)
확인된 항목
- 파일 업로드 드래그앤드롭 UI 존재
- 버전 이력 조회 UI 존재
- 다운로드 흐름 존재
- SHA-256 해시 전제 설명 문구와 버전 표시 UI 존재

판단
- 단순 파일 첨부가 아니라 **무결성/버전 관리**를 차별화 포인트로 삼고 있다.

### 4-7. 점수/경보/연동 상태 관리
확인된 항목
- `score.ts`, `alert.ts`, `integration.ts` store 존재
- 점수 조회/재계산/가중치 변경 API 구조 존재
- 경보 조회/읽음처리/탐지 실행 API 구조 존재
- GitHub/Drive 연동 상태 조회/연결/해제 API 구조 존재

판단
- UI/상태 계층은 이미 확장 기능까지 염두에 두고 짜여 있다.
- 다만 실제 백엔드 collector/webhook 구현 여부는 저장소 일부만 보고는 확정할 수 없다.

## 6. 문서와 실제 상태 사이의 차이

이 부분은 반드시 짚고 넘어가야 한다.

현재 `todo.md`는 매우 유용하지만, 실제 저장소 상태와 100% 동기화되어 있지는 않다.

대표 예시
- TODO에는 `GitHub 리포지토리 생성`이 미완료처럼 남아 있으나, 실제로 저장소는 이미 존재함
- TODO에는 `docker-compose.prod.yml` 관련 항목이 미완료로 남아 있지만, 해당 파일은 저장소에 이미 존재함

판단
- 구현이 늦은 것이 아니라 **문서 갱신이 실제 개발 속도를 따라오지 못한 상태**일 가능성이 높다.
- 지금 시점에서는 기능 개발만큼이나 **문서 정합성 정리**가 필요하다.

## 8. 현재 시점의 리스크

### 리스크 1. 문서 드리프트
저장소 내 TODO와 실제 구현 상태가 일부 어긋난다.
이 상태가 길어지면 발표 직전 판단 실수가 생길 수 있다.

### 리스크 2. “구현 존재”와 “실행 안정성”은 다름
파일과 코드가 존재하는 것과,
실제로 전체 시스템이 안정적으로 붙어 돌아가는 것은 다른 문제다.

### 리스크 3. 외부 연동은 UI/계약만 있고 실제 수집 로직은 미완일 수 있음
스토어와 화면은 보여도,
Webhook / Polling / Collector / OAuth 실제 동작은 아직 덜 붙었을 가능성이 있다.

현재 `Capstone_b`는 다음과 같이 평가할 수 있다.

### 이미 확보한 것
- 프로젝트 목적이 명확함

- 데모 가능한 수준에 근접한 흔적이 많음

### 아직 필요한 것
- 실제 전체 실행 검증
- 데모 안정화
- 문서 정리
- 외부 연동 및 AI 확장 여부 확정

### 결론
이 저장소는 **아이디어 단계가 아니라, “중간발표/데모 직전의 MVP 통합 단계”**로 보는 것이 가장 정확하다.

## 11. 이번 checkpoint 문서의 의미

이 문서는 “지금 어디까지 왔는지”를 정리한 **중간 점검 기준점**이다.

따라서 이후 작업에서는 아래 기준으로 관리하는 것이 좋다.
- `todo.md`: 세부 작업 체크리스트
- `checkpoint.md`: 큰 흐름과 현재 위치 요약
- 모듈 문서: 기능별 세부 설명

이 구조로 가면 발표 준비와 유지보수가 훨씬 쉬워진다.
